## The idea

Pick a pivot. Move every element ≤ pivot to its left, every element > pivot to its right. The pivot is now in its final sorted position; recurse on each side independently.

This is the **Lomuto** partition — simpler than Hoare, slightly worse on duplicates, but the one most interviewers want on a whiteboard.

## Why it's the default

Most production sort routines (`std::sort`, V8's `Array.prototype.sort`, Java's primitive sort) start with quicksort or introsort (quicksort + heapsort fallback). The reasons are:

- **In-place** — no O(n) extra buffer like mergesort.
- **Cache-friendly** — partitioning is sequential, prefetcher loves it.
- **Tail-recursion-friendly** — the smaller half can be processed first to keep stack depth at O(log n) worst-case.

The downside is the O(n²) worst case. Real implementations mitigate it three ways:

1. **Randomized pivot** — break adversarial input patterns.
2. **Median-of-three** — sample three elements, take the median.
3. **Introsort** — fall back to heapsort if recursion depth exceeds 2·log₂n.

## When to reach for something else

- Need **stability** (equal keys keep relative order) → mergesort or Timsort.
- Need **strict O(n log n)** guarantees (e.g. real-time) → heapsort or mergesort.
- Sorting **mostly-sorted** data → Timsort or insertion sort wins easily.
