## The idea

Split the array in half. Sort each half (recursively). Merge the two sorted halves by walking both with two pointers, picking the smaller head each time.

The recursion depth is `log₂n`. Each level does `O(n)` work merging. So total: `O(n log n)`, unconditionally.

## Why it's the right choice when…

- **Stability matters.** Mergesort doesn't reorder equal keys.
- **Worst case matters.** Quicksort can degenerate; mergesort cannot.
- **Sorting linked lists.** Linked-list mergesort is in-place (rewiring pointers), no extra buffer.
- **External sorting.** When data doesn't fit in memory, the merge step generalizes to k-way merging across sorted runs on disk — every database engine uses this for external sort + spill-to-disk.

## Trade-off vs quicksort

| | Mergesort | Quicksort |
|---|-----------|-----------|
| Worst case | O(n log n) | O(n²) |
| Stable | yes | no (Lomuto) |
| Extra space | O(n) | O(log n) |
| Cache locality | poor (alternating reads) | excellent |
| Real-world speed | slower constant factor | usually faster |

See [quicksort](/learn/dsa/quicksort/) for the in-place / faster-constant alternative.

In practice: **Timsort** (Python, Java for objects, V8 since 2018) uses mergesort + insertion-sort for short runs + galloping to combine sorted runs efficiently. It's mergesort, tuned.
