## Problem
Given an integer array, return all **unique** triplets `(a, b, c)` with `a + b + c == 0`. No duplicate triplets in the output.

## Intuition
Sort the array, then reduce 3Sum to 2Sum. Fix `i` as the smallest element of the triplet; then on the suffix `[i+1, n-1]`, run a two-pointer search for `target = -arr[i]`. The sort enables both the two-pointer step and the *cheap* duplicate dedup (just skip equal neighbours). The invariant: once you fix `i`, every valid `(l, r)` is found in a single O(n) sweep, so the whole algorithm is O(n²) — the sort is dominated.

## Algorithm
1. Sort `arr`.
2. For `i` from 0 to `n - 3`:
   - If `arr[i] > 0`, break — no negative + non-negative can equal `-positive`.
   - If `i > 0` and `arr[i] == arr[i - 1]`, **skip** (duplicate triplet root).
   - Set `l = i + 1`, `r = n - 1`, `target = -arr[i]`.
   - While `l < r`:
     - `s = arr[l] + arr[r]`.
     - If `s == target`: record `(arr[i], arr[l], arr[r])`; advance `l` past duplicates and retreat `r` past duplicates.
     - Else if `s < target`: `l++`. Else `r--`.

```python
def three_sum(arr):
    arr.sort()
    out = []
    n = len(arr)
    for i in range(n - 2):
        if arr[i] > 0:
            break
        if i > 0 and arr[i] == arr[i - 1]:
            continue
        l, r, target = i + 1, n - 1, -arr[i]
        while l < r:
            s = arr[l] + arr[r]
            if s == target:
                out.append([arr[i], arr[l], arr[r]])
                while l < r and arr[l] == arr[l + 1]:
                    l += 1
                while l < r and arr[r] == arr[r - 1]:
                    r -= 1
                l += 1
                r -= 1
            elif s < target:
                l += 1
            else:
                r -= 1
    return out
```

## Walkthrough
On sorted `arr = [-4, -1, -1, 0, 1, 2]`:

1. **Anchor `i=0` (`-4`), target = 4.** Two-pointer sweep `l=1, r=5` walks every sum `≤ 3 < 4`, so `l` keeps advancing and no triple is recorded.
2. **Anchor `i=1` (`-1`), target = 1.** `l=2, r=5`: `-1 + 2 = 1` — record `(-1, -1, 2)`. Advance to `l=3, r=4`: `0 + 1 = 1` — record `(-1, 0, 1)`.
3. **Anchor `i=2` (`-1`).** `arr[2] == arr[1]` → skip the duplicate anchor.
4. **Anchor `i=3` (`0`), target = 0.** `l=4, r=5`: `1 + 2 = 3 > 0` → `r--`, pointers cross, no new triple.
5. Final result: `[[-1, -1, 2], [-1, 0, 1]]`.

## Complexity

- time **O(n²)**
- space **O(1)** extra (excluding output)

## Pitfalls
- Forgetting to **sort first** — without sort, you can't skip duplicates cleanly and the two-pointer step doesn't apply.
- Skipping duplicates only on `i` but not on `l` and `r` — you'll emit duplicate triplets.
- Not breaking when `arr[i] > 0` — an easy pruning that halves the constant factor.
- Returning indices instead of values when the problem asks for values, or vice versa.
- Using a hash-set for dedup of triplets — works, but O(n²) extra space; the sort-then-skip approach is cleaner.
