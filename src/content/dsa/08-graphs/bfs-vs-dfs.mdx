## The single algorithm

```ts
function traverse(start: number, adj: number[][]) {
  const seen = new Set([start]);
  const frontier = [start];                 // queue OR stack — same shape
  while (frontier.length) {
    const u = frontier.shift();             // BFS: FIFO
    // const u = frontier.pop();            // DFS: LIFO
    visit(u);
    for (const v of adj[u]) {
      if (!seen.has(v)) {
        seen.add(v);
        frontier.push(v);
      }
    }
  }
}
```

The *only* difference between BFS and DFS is whether you take from the head or the tail of your frontier. Everything else — the visited set, the loop structure, the complexity — is identical.

## Pick by the question being asked

| Question | Reach for |
|----------|----------|
| Shortest path in an unweighted graph | **BFS** (first time you see a node = fewest edges) |
| Detect any cycle | DFS (back-edge during exploration = cycle) |
| Topological sort | DFS (post-order reverse) or Kahn's algorithm (BFS on in-degree 0) |
| Connected components | Either — count component-roots seen |
| "Can I get from A to B?" | Either — same complexity, BFS finds shorter answer |
| Bipartite check | BFS with alternating-layer coloring |
| Find an articulation point / bridge | DFS with discovery / low-link |
| Web crawl, AI search frontier | BFS by default (avoid getting stuck deep in one subtree) |

## The visit-order trap

BFS and DFS visit the same set of nodes — they always discover everything reachable from the start. What changes is the *order*, which matters when:

- You need the **shortest path** (BFS only).
- You're doing **early termination** ("is there *any* node satisfying P?"): the algorithm you pick changes which match it finds first, even though both finish.
- You're computing per-node values that depend on parents (BFS yields parents in non-decreasing distance order; DFS doesn't).

For weighted shortest paths, neither works — see [Dijkstra](/learn/dsa/dijkstra-shortest-path/).
